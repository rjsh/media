117/**
118 * Deinterlace filter. Performs inverse telecine.
119 *
120 * Also known as "film mode" or "3:2 reverse pulldown" in some equipment.
121 *
122 * This filter attempts to reconstruct the original film frames from an
123 * NTSC telecined signal. It is intended for 24fps progressive material
124 * that was telecined to NTSC 60i. For example, most NTSC anime DVDs
125 * are like this.
126 *
127 * There is no input frame parameter, because the input frames
128 * are taken from the history buffer.
129 *
130 * This algorithm does CUSTOM_PTS timestamp mangling.
131 *
132 * See the file comment for a detailed description of the algorithm.
133 *
134 * @param p_filter The filter instance. Must be non-NULL.
135 * @param[out] p_dst Output frame. Must be allocated by caller.
136 * @return VLC error code (int).
137 * @retval VLC_SUCCESS A film frame was reconstructed to p_dst.
138 * @retval VLC_EGENERIC Frame dropped as part of normal IVTC operation.
139 * @see Deinterlace()
140 * @see ComposeFrame()
141 * @see CalculateInterlaceScore()
142 * @see EstimateNumBlocksWithMotion()
143 */
144int RenderIVTC( filter_t *p_filter, picture_t *p_dst );

163/**
164 * \file
165 * IVTC (inverse telecine) algorithm for the VLC deinterlacer.
166 * Also known as "film mode" or "3:2 reverse pulldown" in some equipment.
167 *
168 * Summary:
169 *
170 * This is a "live IVTC" filter, which attempts to do in realtime what
171 * Transcode's ivtc->decimate->32detect chain does offline. Additionally,
172 * it removes soft telecine. It is an original design, based on some ideas
173 * from Transcode, some from TVTime/Xine, and some original.
174 *
175 * If the input material is pure NTSC telecined film, inverse telecine
176 * will (ideally) exactly recover the original progressive film frames.
177 * The output will run at 4/5 of the original framerate with no loss of
178 * information. Interlacing artifacts are removed, and motion becomes
179 * as smooth as it was on the original film. For soft-telecined material,
180 * on the other hand, the progressive frames alredy exist, so only the
181 * timings are changed such that the output becomes smooth 24fps (or would,
182 * if the output device had an infinite framerate).
183 *
184 * Put in simple terms, this filter is targeted for NTSC movies and
185 * especially anime. Virtually all 1990s and early 2000s anime is
186 * hard-telecined. Because the source material is like that,
187 * IVTC is needed for also virtually all official R1 (US) anime DVDs.
188 *
189 * Note that some anime from the turn of the century (e.g. Silent Mobius
190 * and Sol Bianca) is a hybrid of telecined film and true interlaced
191 * computer-generated effects and camera pans. In this case, applying IVTC
192 * will effectively attempt to reconstruct the frames based on the film
193 * component, but even if this is successful, the framerate reduction will
194 * cause the computer-generated effects to stutter. This is mathematically
195 * unavoidable. Instead of IVTC, a framerate doubling deinterlacer is
196 * recommended for such material. Try "Phosphor", "Bob", or "Linear".
197 *
198 * Fortunately, 30fps true progressive anime is on the rise (e.g. ARIA,
199 * Black Lagoon, Galaxy Angel, Ghost in the Shell: Solid State Society,
200 * Mai Otome, Last Exile, and Rocket Girls). This type requires no
201 * deinterlacer at all.
202 *
203 * Another recent trend is using 24fps computer-generated effects and
204 * telecining them along with the cels (e.g. Kiddy Grade, Str.A.In. and
205 * The Third: The Girl with the Blue Eye). For this group, IVTC is the
206 * correct way to deinterlace, and works properly.
207 *
208 * Soft telecined anime, while rare, also exists. Stellvia of the Universe
209 * and Angel Links are examples of this. Stellvia constantly alternates
210 * between soft and hard telecine - pure CGI sequences are soft-telecined,
211 * while sequences incorporating cel animation are hard-telecined.
212 * This makes it very hard for the cadence detector to lock on,
213 * and indeed Stellvia gives some trouble for the filter.
214 *
215 * To finish the list of different material types, Azumanga Daioh deserves
216 * a special mention. The OP and ED sequences are both 30fps progressive,
217 * while the episodes themselves are hard-telecined. This filter should
218 * mostly work correctly with such material, too. (The beginning of the OP
219 * shows some artifacts, but otherwise both the OP and ED are indeed
220 * rendered progressive. The technical reason is that the filter has been
221 * designed to aggressively reconstruct film frames, which helps in many
222 * cases with hard-telecined material. In very rare cases, this approach may
223 * go wrong, regardless of whether the input is telecined or progressive.)
224 *
225 * Finally, note also that IVTC is the only correct way to deinterlace NTSC
226 * telecined material. Simply applying an interpolating deinterlacing filter
227 * (with no framerate doubling) is harmful for two reasons. First, even if
228 * the filter does not damage already progressive frames, it will lose half
229 * of the available vertical resolution of those frames that are judged
230 * interlaced. Some algorithms combining data from multiple frames may be
231 * able to counter this to an extent, effectively performing something akin
232 * to the frame reconstruction part of IVTC. A more serious problem is that
233 * any motion will stutter, because (even in the ideal case) one out of
234 * every four film frames will be shown twice, while the other three will
235 * be shown only once. Duplicate removal and framerate reduction - which are
236 * part of IVTC - are also needed to properly play back telecined material
237 * on progressive displays at a non-doubled framerate.
238 *
239 * So, try this filter on your NTSC anime DVDs. It just might help.
240 *
241 *
242 * Technical details:
243 *
244 *
245 * First, NTSC hard telecine in a nutshell:
246 *
247 * Film is commonly captured at 24 fps. The framerate must be raised from
248 * 24 fps to 59.94 fields per second, This starts by pretending that the
249 * original framerate is 23.976 fps. When authoring, the audio can be
250 * slowed down by 0.1% to match. Now 59.94 = 5/4 * (2*23.976), which gives
251 * a nice ratio made out of small integers.
252 *
253 * Thus, each group of four film frames must become five frames in the NTSC
254 * video stream. One cannot simply repeat one frame of every four, because
255 * this would result in jerky motion. To slightly soften the jerkiness,
256 * the extra frame is split into two extra fields, inserted at different
257 * times. The content of the extra fields is (in classical telecine)
258 * duplicated as-is from existing fields.
259 *
260 * The field duplication technique is called "3:2 pulldown". The pattern
261 * is called the cadence. The output from 3:2 pulldown looks like this
262 * (if the telecine is TFF, top field first):
263 *
264 * a b c d e Telecined frame (actual frames stored on DVD)
265 * T1 T1 T2 T3 T4 *T*op field content
266 * B1 B2 B3 B3 B4 *B*ottom field content
267 *
268 * Numbers 1-4 denote the original film frames. E.g. T1 = top field of
269 * original film frame 1. The field Tb, and one of either Bc or Bd, are
270 * the extra fields inserted in the telecine. With exact duplication, it
271 * of course doesn't matter whether Bc or Bd is the extra field, but
272 * with "full field blended" material (see below) this will affect how to
273 * correctly extract film frame 3.
274 *
275 * See the following web pages for illustrations and discussion:
276 * http://neuron2.net/LVG/telecining1.html
277 * http://arbor.ee.ntu.edu.tw/~jackeikuo/dvd2avi/ivtc/
278 *
279 * Note that film frame 2 has been stored "half and half" into two telecined
280 * frames (b and c). Note also that telecine produces a sequence of
281 * 3 progressive frames (d, e and a) followed by 2 interlaced frames
282 * (b and c).
283 *
284 * The output may also look like this (BFF telecine, bottom field first):
285 *
286 * a' b' c' d' e'
287 * T1 T2 T3 T3 T4
288 * B1 B1 B2 B3 B4
289 *
290 * Now field Bb', and one of either Tc' or Td', are the extra fields.
291 * Again, film frame 2 is stored "half and half" (into b' and c').
292 *
293 * Whether the pattern is like abcde or a'b'c'd'e', depends on the telecine
294 * field dominance (TFF or BFF). This must match the video field dominance,
295 * but is conceptually different. Importantly, there is no temporal
296 * difference between those fields that came from the same film frame.
297 * Also, see the section on soft telecine below.
298 *
299 * In a hard telecine, the TFD and VFD must match for field renderers
300 * (e.g. traditional DVD player + CRT TV) to work correctly; this should be
301 * fairly obvious by considering the above telecine patterns and how a
302 * field renderer displays the material (one field at a time, dominant
303 * field first).
304 *
305 * The VFD may, *correctly*, flip mid-stream, if soft field repeats
306 * (repeat_pict) have been used. They are commonly used in soft telecine
307 * (see below), but also occasional lone field repeats exist in some streams,
308 * e.g., Sol Bianca.
309 *
310 * See e.g.
311 * http://www.cambridgeimaging.co.uk/downloads/Telecine%20field%20dominance.pdf
312 * for discussion. The document discusses mostly PAL, but includes some notes
313 * on NTSC, too.
314 *
315 * The reason for the words "classical telecine" above, when field
316 * duplication was first mentioned, is that there exists a
317 * "full field blended" version, where the added fields are not exact
318 * duplicates, but are blends of the original film frames. This is rare
319 * in NTSC, but some material like this reportedly exists. See
320 * http://www.animemusicvideos.org/guides/avtech/videogetb2a.html
321 * In these cases, the additional fields are a (probably 50%) blend of the
322 * frames between which they have been inserted. Which one of the two
323 * possibilites is the extra field then becomes important.
324 * This filter does NOT support "full field blended" material.
325 *
326 * To summarize, the 3:2 pulldown sequence produces a group of ten fields
327 * out of every four film frames. Only eight of these fields are unique.
328 * To remove the telecine, the duplicate fields must be removed, and the
329 * original progressive frames restored. Additionally, the presentation
330 * timestamps (PTS) must be adjusted, and one frame out of five (containing
331 * no new information) dropped. The duration of each frame in the output
332 * becomes 5/4 of that in the input, i.e. 25% longer.
333 *
334 * Theoretically, this whole mess could be avoided by soft telecining, if the
335 * original material is pure 24fps progressive. By using the stream flags
336 * correctly, the original progressive frames can be stored on the DVD.
337 * In such cases, the DVD player will apply "soft" 3:2 pulldown. See the
338 * following section.
339 *
340 * Also, the mess with cadence detection for hard telecine (see below) could
341 * be avoided by using the progressive frame flag and a five-frame future
342 * buffer, but no one ever sets the flag correctly for hard-telecined
343 * streams. All frames are marked as interlaced, regardless of their cadence
344 * position. This is evil, but sort-of-understandable, given that video
345 * editors often come with "progressive" and "interlaced" editing modes,
346 * but no separate "telecined" mode that could correctly handle this
347 * information.
348 *
349 * In practice, most material with its origins in Asia (including virtually
350 * all official US (R1) anime DVDs) is hard-telecined. Combined with the
351 * turn-of-the-century practice of rendering true interlaced effects
352 * on top of the hard-telecined stream, we have what can only be described
353 * as a monstrosity. Fortunately, recent material is much more consistent,
354 * even though still almost always hard-telecined.
355 *
356 * Finally, note that telecined video is often edited directly in interlaced
357 * form, disregarding safe cut positions as pertains to the telecine sequence
358 * (there are only two: between "d" and "e", or between "e" and the
359 * next "a"). Thus, the telecine sequence will in practice jump erratically
360 * at cuts [**]. An aggressive detection strategy is needed to cope with
361 * this.
362 *
363 * [**] http://users.softlab.ece.ntua.gr/~ttsiod/ivtc.html
364 *
365 *
366 * Note about chroma formats: 4:2:0 is very common at least on anime DVDs.
367 * In the interlaced frames in a hard telecine, the chroma alternates
368 * every chroma line, even if the chroma format is 4:2:0! This means that
369 * if the interlaced picture is viewed as-is, the luma alternates every line,
370 * while the chroma alternates only every two lines of the picture.
371 *
372 * That is, an interlaced frame in a 4:2:0 telecine looks like this
373 * (numbers indicate which film frame the data comes from):
374 *
375 * luma stored 4:2:0 chroma displayed chroma
376 * 1111 1111 1111
377 * 2222 1111
378 * 1111 2222 2222
379 * 2222 2222
380 * ... ... ...
381 *
382 * The deinterlace filter sees the stored 4:2:0 chroma. The "displayed chroma"
383 * is only generated later in the filter chain (probably when YUV is converted
384 * to the display format, if the display does not accept YUV 4:2:0 directly).
385 *
386 *
387 * Next, how NTSC soft telecine works:
388 *
389 * a b c d Frame index (actual frames stored on DVD)
390 * T1 T2 T3 T4 *T*op field content
391 * B1 B2 B3 B4 *B*ottom field content
392 *
393 * Here the progressive frames are stored as-is. The catch is in the stream
394 * flags. For hard telecine, which was explained above, we have
395 * VFD = constant and nb_fields = 2, just like in a true progressive or
396 * true interlaced stream. Soft telecine, on the other hand, looks like this:
397 *
398 * a b c d
399 * 3 2 3 2 nb_fields
400 * T B B T *Video* field dominance (for TFF telecine)
401 * B T T B *Video* field dominance (for BFF telecine)
402 *
403 * Now the video field dominance flipflops every two frames!
404 *
405 * Note that nb_fields = 3 means the frame duration will be 1.5x that of a
406 * normal frame. Often, soft-telecined frames are correctly flagged as
407 * progressive.
408 *
409 * Here the telecining is expected to be done by the player, utilizing the
410 * soft field repeat (repeat_pict) feature. This is indeed what a field
411 * renderer (traditional interlaced equipment, or a framerate doubler)
412 * should do with such a stream.
413 *
414 * In the IVTC filter, our job is to even out the frame durations, but
415 * disregard video field dominance and just pass the progressive pictures
416 * through as-is.
417 *
418 * Fortunately, for soft telecine to work at all, the stream flags must be
419 * set correctly. Thus this type can be detected reliably by reading
420 * nb_fields from three consecutive frames:
421 *
422 * Let P = previous, C = current, N = next. If the frame to be rendered is C,
423 * there are only three relevant nb_fields flag patterns for the three-frame
424 * stencil concerning soft telecine:
425 *
426 * P C N What is happening:
427 * 2 3 2 Entering soft telecine at frame C, or running inside it already.
428 * 3 2 3 Running inside soft telecine.
429 * 3 2 2 Exiting soft telecine at frame C. C is the last frame that should
430 * be handled as soft-telecined. (If we do timing adjustments to the
431 * "3"s only, we can already exit soft telecine mode when we see
432 * this pattern.)
433 *
434 * Note that the same stream may alternate between soft and hard telecine,
435 * but these cannot occur at the same time. The start and end of the
436 * soft-telecined parts can be read off the stream flags, and the rest of
437 * the stream can be handed to the hard IVTC part of the filter for analysis.
438 *
439 * Finally, note also that a stream may also request a lone field repeat
440 * (a sudden "3" surrounded by "2"s). Fortunately, these can be handled as
441 * a two-frame soft telecine, as they match the first and third
442 * flag patterns above.
443 *
444 * Combinations with several "3"s in a row are not valid for soft or hard
445 * telecine, so if they occur, the frames can be passed through as-is.
446 *
447 *
448 * Cadence detection for hard telecine:
449 *
450 * Consider viewing the TFF and BFF hard telecine sequences through a
451 * three-frame stencil. Again, let P = previous, C = current, N = next.
452 * A brief analysis leads to the following cadence tables.
453 *
454 * PCN = stencil position (Previous Current Next),
455 * Dups. = duplicate fields,
456 * Best field pairs... = combinations of fields which correctly reproduce
457 * the original progressive frames,
458 * * = see timestamp considerations below for why
459 * this particular arrangement.
460 *
461 * For TFF:
462 *
463 * PCN Dups. Best field pairs for progressive (correct, theoretical)
464 * abc TP = TC TPBP = frame 1, TCBP = frame 1, TNBC = frame 2
465 * bcd BC = BN TCBP = frame 2, TNBC = frame 3, TNBN = frame 3
466 * cde BP = BC TCBP = frame 3, TCBC = frame 3, TNBN = frame 4
467 * dea none TPBP = frame 3, TCBC = frame 4, TNBN = frame 1
468 * eab TC = TN TPBP = frame 4, TCBC = frame 1, TNBC = frame 1
469 *
470 * (table cont'd)
471 * PCN Progressive output*
472 * abc frame 2 = TNBC (compose TN+BC)
473 * bcd frame 3 = TNBN (copy N)
474 * cde frame 4 = TNBN (copy N)
475 * dea (drop)
476 * eab frame 1 = TCBC (copy C), or TNBC (compose TN+BC)
477 *
478 * On the rows "dea" and "eab", frame 1 refers to a frame from the next
479 * group of 4. "Compose TN+BC" means to construct a frame using the
480 * top field of N, and the bottom field of C. See ComposeFrame().
481 *
482 * For BFF, swap all B and T, and rearrange the symbol pairs to again
483 * read "TxBx". We have:
484 *
485 * PCN Dups. Best field pairs for progressive (correct, theoretical)
486 * abc BP = BC TPBP = frame 1, TPBC = frame 1, TCBN = frame 2
487 * bcd TC = TN TPBC = frame 2, TCBN = frame 3, TNBN = frame 3
488 * cde TP = TC TPBC = frame 3, TCBC = frame 3, TNBN = frame 4
489 * dea none TPBP = frame 3, TCBC = frame 4, TNBN = frame 1
490 * eab BC = BN TPBP = frame 4, TCBC = frame 1, TCBN = frame 1
491 *
492 * (table cont'd)
493 * PCN Progressive output*
494 * abc frame 2 = TCBN (compose TC+BN)
495 * bcd frame 3 = TNBN (copy N)
496 * cde frame 4 = TNBN (copy N)
497 * dea (drop)
498 * eab frame 1 = TCBC (copy C), or TCBN (compose TC+BN)
499 *
500 * From these cadence tables we can extract two strategies for
501 * cadence detection. We use both.
502 *
503 * Strategy 1: duplicated fields ("vektor").
504 *
505 * Consider that each stencil position has a unique duplicate field
506 * condition. In one unique position, "dea", there is no match; in all
507 * other positions, exactly one. By conservatively filtering the
508 * possibilities based on detected hard field repeats (identical fields
509 * in successive input frames), it is possible to gradually lock on
510 * to the cadence. This kind of strategy is used by the classic IVTC filter
511 * in TVTime/Xine by Billy Biggs (Vektor), hence the name.
512 *
513 * "Conservative" here means that we do not rule anything out, but start at
514 * each stencil position by suggesting the position "dea", and then only add
515 * to the list of possibilities based on field repeats that are detected at
516 * the present stencil position. This estimate is then filtered by ANDing
517 * against a shifted (time-advanced) version of the estimate from the
518 * previous stencil position. Once the detected position becomes unique,
519 * the filter locks on. If the new detection is inconsistent with the
520 * previous one, the detector resets itself and starts from scratch.
521 *
522 * The strategy is very reliable, as it only requires running (fuzzy)
523 * duplicate field detection against the input. It is very good at staying
524 * locked on once it acquires the cadence, and it does so correctly very
525 * often. These are indeed characteristics that can be observed in the
526 * behaviour of the TVTime/Xine filter.
527 *
528 * Note especially that 8fps/12fps animation, common in anime, will cause
529 * spurious hard-repeated fields. The conservative nature of the method
530 * makes it very good at dealing with this - any spurious repeats will only
531 * slow down the lock-on, not completely confuse it. It should also be good
532 * at detecting the presence of a telecine, as neither true interlaced nor
533 * true progressive material should contain any hard field repeats.
534 * (This, however, has not been tested yet.)
535 *
536 * The disadvantages are that at times the method may lock on slowly,
537 * because the detection must be filtered against the history until
538 * a unique solution is found. Resets, if they happen, will also
539 * slow down the lock-on.
540 *
541 * The hard duplicate detection required by this strategy can be made
542 * data-adaptive in several ways. TVTime uses a running average of motion
543 * scores for its history buffer. We utilize a different, original approach.
544 * It is rare, if not nonexistent, that only one field changes between
545 * two valid frames. Thus, if one field changes "much more" than the other
546 * in fieldwise motion detection, the less changed one is probably a
547 * duplicate. Importantly, this works with telecined input, too - the field
548 * that changes "much" may be part of another film frame, while the "less"
549 * changed one is actually a duplicate from the previous film frame.
550 * If both fields change "about as much", then no hard field repeat
551 * is detected.
552 *
553 *
554 * Strategy 2: progressive/interlaced field combinations ("scores").
555 *
556 * We can also form a second strategy, which is not as reliable in practice,
557 * but which locks on faster when it does. This is original to this filter.
558 *
559 * Consider all possible field pairs from two successive frames: TCBC, TCBN,
560 * TNBC, TNBN. After one frame, these become TPBP, TPBC, TCBP, TCBC.
561 * These eight pairs (seven unique, disregarding the duplicate TCBC)
562 * are the exhaustive list of possible field pairs from two successive
563 * frames in the three-frame PCN stencil.
564 *
565 * The above tables list triplets of field pair combinations for each cadence
566 * position, which should produce progressive frames. All the given triplets
567 * are unique in each table alone, although the one at "dea" is
568 * indistinguishable from the case of pure progressive material. It is also
569 * the only one which is not unique across both tables.
570 *
571 * Thus, all sequences of two neighboring triplets are unique across both
572 * tables. (For "neighboring", each table is considered to wrap around from
573 * "eab" back to "abc", i.e. from the last row back to the first row.)
574 * Furthermore, each sequence of three neighboring triplets is redundantly
575 * unique (i.e. is unique, and reduces the chance of false positives).
576 * (In practice, though, we already know which table to consider, from the fact
577 * that TFD and VFD must match. Checking only the relevant table makes the
578 * strategy slightly more robust.)
579 *
580 * The important idea is: *all other* field pair combinations should produce
581 * frames that look interlaced. This includes those combinations present in
582 * the "wrong" (i.e. not current position) rows of the table (insofar as
583 * those combinations are not also present in the "correct" row; by the
584 * uniqueness property, *every* "wrong" row will always contain at least one
585 * combination that differs from those in the "correct" row).
586 *
587 * We generate the artificial frames TCBC, TCBN, TNBC and TNBN (virtually;
588 * no data is actually moved). Two of these are just the frames C and N,
589 * which already exist; the two others correspond to composing the given
590 * field pairs. We then compute the interlace score for each of these frames.
591 * The interlace scores of what are now TPBP, TPBC and TCBP, also needed,
592 * were computed by this same mechanism during the previous input frame.
593 * These can be slided in history and reused.
594 *
595 * We then check, using the computed interlace scores, and taking into
596 * account the video field dominance information, which field combination
597 * triplet given in the appropriate table produces the smallest sum of
598 * interlace scores. Unless we are at PCN = "dea" (which could also be pure
599 * progressive!), this immediately gives us the most likely current cadence
600 * position. Combined with a two-step history, the sequence of three most
601 * likely positions found this way always allows us to make a more or less
602 * reliable detection. (That is, when a reliable detection is possible; if the
603 * video has no motion at all, every detection will report the position "dea".
604 * In anime, still shots are common. Thus we must augment this with a
605 * full-frame motion detection that switches the detector off if no motion
606 * was detected.)
607 *
608 * The detection seems to need four full-frame interlace analyses per frame.
609 * Actually, three are enough, because the previous N is the new C, so we can
610 * slide the already computed result. Also during initialization, we only
611 * need to compute TNBN on the first frame; this has become TPBP when the
612 * third frame is reached. Similarly, we compute TNBN, TNBC and TCBN during
613 * the second frame (just before the filter starts), and these get slided
614 * into TCBC, TCBP and TPBC when the third frame is reached. At that point,
615 * initialization is complete.
616 *
617 * Because we only compare interlace scores against each other, no threshold
618 * is needed in the cadence detector. Thus it, trivially, adapts to the
619 * material automatically.
620 *
621 * The weakness of this approach is that any comb metric detects incorrectly
622 * every now and then. Especially slow vertical camera pans often get treated
623 * wrong, because the messed-up field combination looks less interlaced
624 * according to the comb metric (especially in anime) than the correct one
625 * (which contains, correctly, one-pixel thick cartoon outlines, parts of
626 * which often perfectly horizontal).
627 *
628 * The advantage is that this strategy catches horizontal camera pans
629 * immediately and reliably, while the other strategy may still be trying
630 * to lock on.
631 *
632 *
633 * Frame reconstruction:
634 *
635 * We utilize a hybrid approach. If a valid cadence is locked on, we use the
636 * operation table to decide what to do. This handles those cases correctly,
637 * which would be difficult for the interlace detector alone (e.g. vertical
638 * camera pans). Note that the operations that must be performed for IVTC
639 * include timestamp mangling and frame dropping, which can only be done
640 * reliably on a valid cadence.
641 *
642 * When the cadence fails (we detect this from a sudden upward jump in the
643 * interlace scores of the constructed frames), we reset the "vektor"
644 * detector strategy and fall back to an emergency frame composer, where we
645 * use ideas from Transcode's IVTC.
646 *
647 * In this emergency mode, we simply output the least interlaced frame out of
648 * the combinations TNBN, TNBC and TCBN (where only one of the last two is
649 * tested, based on the stream TFF/BFF information). In this mode, we do not
650 * touch the timestamps, and just pass all five frames from each group right
651 * through. This introduces some stutter, but in practice it is often not
652 * noticeable. This is because the kind of material that is likely to trip up
653 * the cadence detector usually includes irregular 8fps/12fps motion. With
654 * true 24fps motion, the cadence quickly locks on, and stays locked on.
655 *
656 * Once the cadence locks on again, we resume normal operation based on
657 * the operation table.
658 *
659 *
660 * Timestamp mangling:
661 *
662 * To make five into four we need to extend frame durations by 25%.
663 * Consider the following diagram (times given in 90kHz ticks, rounded to
664 * integers; this is just for illustration, and for comparison with the
665 * "scratch paper" comments in pulldown.c of TVTime/Xine):
666 *
667 * NTSC input (29.97 fps)
668 * a b c d e a (from next group) ...
669 * 0 3003 6006 9009 12012 15015
670 * 0 3754 7508 11261 15015
671 * 1 2 3 4 1 (from next group) ...
672 * Film output (23.976 fps)
673 *
674 * Three of the film frames have length 3754, and one has 3753
675 * (it is 1/90000 sec shorter). This rounding was chosen so that the lengths
676 * of the group of four sum to the original 15015.
677 *
678 * From the diagram we get these deltas for presentation timestamp adjustment
679 * (in 90 kHz ticks, for illustration):
680 * (1-a) (2-b) (3-c) (4-d) (skip) (1-a) ...
681 * 0 +751 +1502 +2252 (skip) 0 ...
682 *
683 * In fractions of (p_next->date - p_cur->date), regardless of actual
684 * time unit, the deltas are:
685 * (1-a) (2-b) (3-c) (4-d) (skip) (1-a) ...
686 * 0 +0.25 +0.50 +0.75 (skip) 0 ...
687 *
688 * This is what we actually use. In our implementation, the values are stored
689 * multiplied by 4, as integers.
690 *
691 * The "current" frame should be displayed at [original time + delta].
692 * E.g., when "current" = b (i.e. PCN = abc), start displaying film frame 2
693 * at time [original time of b + 751 ticks]. So, when we catch the cadence,
694 * we will start mangling the timestamps according to the cadence position
695 * of the "current" frame, using the deltas given above. This will cause
696 * a one-time jerk, most noticeable if the cadence happens to catch at
697 * position "d". (Alternatively, upon lock-on, we could wait until we are
698 * at "a" before switching on IVTC, but this makes the maximal delay
699 * [max. detection + max. wait] = 3 + 4 = 7 input frames, which comes to
700 * 7/30 ~ 0.23 seconds instead of the 3/30 = 0.10 seconds from purely
701 * the detection. The one-time jerk is simpler to implement and gives the
702 * faster lock-on.)
703 *
704 * It is clear that "e" is a safe choice for the dropped frame. This can be
705 * seen from the timings and the cadence tables. First, consider the timings.
706 * If we have only one future frame, "e" is the only one whose PTS, comparing
707 * to the film frames, allows dropping it safely. To see this, consider which
708 * film frame needs to be rendered as each new input frame arrives. Secondly,
709 * consider the cadence tables. It is ok to drop "e", because the same
710 * film frame "1" is available also at the next PCN position "eab".
711 * (As a side note, it is interesting that Vektor's filter drops "b".
712 * See the TVTime sources.)
713 *
714 * When the filter falls out of film mode, the timestamps of the incoming
715 * frames are left untouched. Thus, the output from this filter has a
716 * variable framerate: 4/5 of the input framerate when IVTC is active
717 * (whether hard or soft), and the same framerate as input when it is not
718 * (or when in emergency mode).
719 *
720 *
721 * For other open-source IVTC codes, which may be a useful source for ideas,
722 * see the following:
723 *
724 * The classic filter by Billy Biggs (Vektor). Written in 2001-2003 for
725 * TVTime, and adapted into Xine later. In xine-lib 1.1.19, it is at
726 * src/post/deinterlace/pulldown.*. Also needed are tvtime.*, and speedy.*.
727 *
728 * Transcode's ivtc->decimate->32detect chain by Thanassis Tsiodras.
729 * Written in 2002, added in Transcode 0.6.12. This probably has something
730 * to do with the same chain in MPlayer, considering that MPlayer acquired
731 * an IVTC filter around the same time. In Transcode 1.1.5, the IVTC part is
732 * at filter/filter_ivtc.c. Transcode 1.1.5 sources can be downloaded from
733 * http://developer.berlios.de/project/showfiles.php?group_id=10094
734 */
